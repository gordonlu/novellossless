## 0. 摘要

novellossless 是一款面向长篇小说作者的本地优先桌面应用。

它不是“替作者写小说”的 AI 写作工具，而是一个帮助作者在长篇创作中守住记忆、设定、伏笔、时间线、人物认知、情绪节奏、考据细节和改稿影响的创作控制系统。

它的核心目标不是让作者写得更多，而是让长篇写作更稳、更连贯、更不容易崩。

novellossless 的第一原则：

1. 不连 AI 也能用。
2. 默认本地运行，不上传正文。
3. AI 只是增强，不是基础依赖。
4. 所有提醒必须有证据来源。
5. 不替作者自动改文，只提供提醒、证据、任务和上下文。
6. 面向普通用户，从第一版开始就是多平台桌面产品。
7. 底层借鉴 deeplossless 的无损上下文、版本记忆、事件溯源、DAG 压缩与本地持久化思想，但小说正文必须有独立完整原文层，不能把压缩摘要当成唯一数据源。

## 1. 产品定位

### 1.1 一句话定位

**novellossless 是长篇小说作者的本地创作记忆与创作控制助手。**

它帮助作者持续记住并控制：

- 人物
- 地点
- 物件
- 事件
- 伏笔
- 暗线
- 设定
- 时间线
- 人物认知边界
- 情绪节奏
- 战力/财富/身份阶梯
- 历史考据
- 世界观规则
- 改稿历史
- 版本漂移
- 读者反馈
- AI 续写上下文

### 1.2 产品口号

> 你写，我记。你改，我追踪。你忘，我提醒。

### 1.3 更深层的产品内核

作者真正害怕的不是“遗忘”，而是“失控”。

不同类型作者害怕的失控不同：

| 作者类型 | 最怕的失控 |
|---|---|
| 长篇作者 | 人物、伏笔、时间线、设定失控 |
| 爽文作者 | 爽点、节奏、战力、读者情绪失控 |
| 历史作者 | 考据、制度、器物、时代细节失控 |
| 悬疑作者 | 线索、公平性、人物认知边界失控 |
| 言情作者 | 情感推进、关系张力、人物动机失控 |
| 科幻作者 | 技术规则、因果链、世界观逻辑失控 |
| 群像作者 | 视角权重、人物状态、关系网络失控 |
| 编辑/工作室 | 审稿证据、问题追踪、修订闭环失控 |

novellossless 要做的不是一个固定功能清单，而是一个可扩展的创作控制面。

## 2. 产品原则

### 2.1 本地优先

默认情况下，所有数据保存在用户本机。

本地保存内容包括：

- 项目配置
- 小说正文索引
- 正文片段
- 章节结构
- 人物卡
- 地点卡
- 物件卡
- 事件卡
- 伏笔卡
- 暗线卡
- 设定规则
- 时间线
- 冲突报告
- 改稿历史
- 修订任务
- 上下文包
- AI 调用记录，若用户启用 AI

未经用户明确授权，不上传小说正文。

### 2.2 离线可用

novellossless 必须在断网、不登录、不配置 AI 的情况下可用。

离线可用能力包括：

- 创建项目
- 导入项目
- 扫描本地文件
- 章节识别
- 全文搜索
- 人物/地点/物件候选
- 伏笔候选
- 基础时间线
- 基础冲突检查
- 重复描写检查
- 改稿记录
- 记忆卡片编辑
- 上下文包导出
- 报告导出
- 隐私中心
- 本地备份

### 2.3 AI 可选增强

AI 不是必需依赖。

默认：

- 不启用 AI。
- 不上传正文。
- 不自动调用云端模型。
- 不要求用户配置 API Key。
- 不阻塞核心功能。

开启 AI 后可增强：

- 隐性伏笔识别
- 复杂因果分析
- 角色认知推理
- 考据风险解释
- 叙事 Diff 解释
- 上下文包润色
- AI 续写前上下文压缩

### 2.4 证据优先

所有提醒必须带来源证据。

错误示例：

```text
林澈设定可能有问题。
```

正确示例：

```text
人物事实冲突：林澈眼睛颜色

证据 A：
第 3 章：林澈抬起黑色的眼睛。

证据 B：
第 18 章：林澈灰蓝色的眼睛在雨里显得冰冷。

请确认：
- 保留旧设定
- 接受新设定
- 标记为伪装
- 标记为角色认知
- 标记为误报
```

### 2.5 不越权改文

novellossless 默认不自动修改用户正文。

系统可以：

- 提醒问题
- 展示证据
- 解释风险
- 创建修订任务
- 生成上下文包
- 生成报告
- 提供建议

系统不能：

- 默认替用户改正文
- 默认覆盖原文件
- 默认删除原文
- 默认上传内容
- 默认读取项目目录外文件

### 2.6 低打扰

novellossless 不是 IDE 编译器，不应把小说写作变成满屏红线。

提醒等级：

| 等级 | 含义 | 展示方式 |
|---|---|---|
| 严重 | 明显破坏设定、时间线、核心事实 | 项目首页高亮，侧边栏置顶 |
| 高 | 可能影响阅读体验或后文逻辑 | 冲突报告醒目展示 |
| 中 | 需要作者确认的疑似问题 | 折叠卡片 |
| 低 | 重复表达、轻微风格漂移 | 周报/报告展示 |
| 信息 | 新人物、新地点、新物件、新伏笔 | 静默记录，可查看 |

### 2.7 用户反馈会改变系统行为

每条提醒都允许用户处理：

- 未处理
- 已确认
- 已解决
- 误报
- 有意为之
- 长期保留
- 稍后处理
- 已废弃

系统必须记住用户反馈。  
如果用户多次将某类提醒标记为“有意为之”或“误报”，后续同类提醒应降低等级或静默记录。

## 3. 目标用户

### 3.1 长篇网文作者

痛点：

- 日更压力大。
- 字数很长。
- 人物、伏笔、设定越来越多。
- 容易忘前文。
- 改前文后不知道哪里受影响。

价值：

- 自动维护人物和伏笔。
- 发现人物设定冲突。
- 提醒长期未回收伏笔。
- 生成下一章上下文包。
- 支持爽文节奏和读者反馈分析。

### 3.2 爽文作者

痛点：

- 连续几章节奏变平，作者自己不易察觉。
- 战力、财富、身份等级容易写崩。
- 高频套路场景需要复用和提纯。
- 读者反馈散落在本章说/评论里。

价值：

- 爽点密度曲线。
- 冲突频次监控。
- 升级间隔提醒。
- 打脸强度分析。
- 战力体系检查。
- 套路模板复用。
- 本章说/评论导入与段落关联。

### 3.3 历史/考据型作者

痛点：

- 官职、地名、器物、食物、货币、制度容易穿帮。
- 架空和真实历史边界难以管理。
- 暗线和互文复杂。
- 人物认知边界严格。

价值：

- 历史框架设定。
- 本地知识包检查。
- 考据风险报告。
- 架空例外机制。
- 暗线网络。
- 人物认知档案。
- 草稿叙事 Diff。

### 3.4 悬疑/推理作者

痛点：

- 线索是否公平很难控制。
- 角色不该知道的信息提前暴露。
- 时间线、证词、物件流转复杂。
- 误导线索和真实线索容易混乱。

价值：

- 线索账本。
- 证词矛盾检查。
- 人物认知边界。
- 时间线严格模式。
- 嫌疑人状态追踪。
- 暗线回响检查。

### 3.5 言情/情感向作者

痛点：

- 关系推进太快或太慢。
- 情绪拉扯重复。
- 人物动机前后不一。
- 误会、告白、和解节点难以平衡。

价值：

- 情感曲线。
- 关系状态追踪。
- 情绪重复检查。
- 关键关系节点提醒。
- 人物动机变化检测。

### 3.6 科幻/设定向作者

痛点：

- 技术规则容易被后文破坏。
- 世界观设定复杂。
- 因果链和限制条件需要严密。
- 设备、星球、组织、科技名词繁多。

价值：

- 技术规则卡。
- 世界观不变量检查。
- 设备状态追踪。
- 组织关系图。
- 因果链检查。

### 3.7 编辑/工作室

痛点：

- 长篇审稿成本高。
- 问题需要证据。
- 多人协作下设定容易混乱。
- 需要输出可读报告。

价值：

- 一致性审查报告。
- 伏笔账本报告。
- 时间线报告。
- 改稿影响报告。
- 章节风险报告。
- Markdown/HTML/PDF 导出。

## 4. 核心产品闭环

novellossless 的核心闭环：

```text
导入小说项目
→ 本地扫描
→ 建立章节、全文索引和记忆卡片
→ 生成伏笔账本、时间线、冲突报告和创作指标
→ 用户确认、忽略、合并、修正或创建修订任务
→ 文件继续变化
→ 增量扫描
→ 发现新问题、改稿影响或节奏变化
→ 更新记忆图和任务状态
→ 导出上下文包、报告或 AI 续写准备材料
```

核心目标是让作者持续知道：

1. 我写过什么。
2. 我改过什么。
3. 我忘了什么。
4. 哪里可能前后冲突。
5. 哪些承诺还没兑现。
6. 哪些设定正在漂移。
7. 当前节奏是否失控。
8. 当前角色知道什么、不知道什么。
9. 下一章需要带上哪些上下文。

## 5. 产品形态

### 5.1 多平台桌面端

首发平台：

1. Windows
2. macOS
3. Linux

优先级：

```text
Windows > macOS > Linux
```

原因：

- 国内普通作者 Windows 用户最多。
- macOS 用户创作软件付费意愿较高。
- Linux 对开发者和开源社区有价值。

### 5.2 推荐技术形态

建议架构：

```text
Tauri / 类 Tauri 桌面壳
前端：Web UI
后端：Rust Core
存储：SQLite
文件监听：原生 watcher
搜索：SQLite FTS5 / Tantivy 可选
AI：可插拔 Provider
```

### 5.3 安装格式

| 平台 | 格式 |
|---|---|
| Windows | `.exe` / `.msi` |
| macOS | `.dmg` |
| Linux | `.AppImage`，可选 `.deb` |

普通用户不应手动启动服务、配置数据库或运行命令行。

### 5.4 不做完整写作编辑器

novellossless 第一阶段不替代 WPS、Word、Typora、VS Code、小黑屋等写作软件。

它负责：

- 项目导入
- 文件监听
- 本地扫描
- 记忆管理
- 一致性审查
- 伏笔管理
- 节奏分析
- 考据检查
- 上下文包导出
- 报告生成

应用内提供：

- 正文浏览
- 来源证据定位
- 卡片编辑
- 备注
- 修订任务
- 报告查看

但不以富文本写作为核心。

## 6. 国内用户适配

### 6.1 文件格式

P0：

- `.txt`
- `.md`

P1：

- `.docx`
- 常见国产写作软件导出的纯文本目录
- GBK / GB18030 / UTF-8 编码自动识别

P2：

- `.html`
- `.epub`
- 其他导出格式

### 6.2 章节识别

必须支持：

```text
第1章
第一章
第001章
第十二章 雨夜
卷一
楔子
序章
番外
Chapter 1
001-雨夜.txt
```

用户可以手动调整章节顺序。

### 6.3 项目结构

必须支持：

```text
单文件长篇
多文件章节目录
正文目录 + 设定目录
正文目录 + 草稿目录
正文、设定、大纲混合目录
```

用户可以手动标记文件类型：

- 正文
- 设定
- 大纲
- 草稿
- 废稿
- 参考资料
- 忽略

### 6.4 中文文本处理

必须针对中文长文本优化：

1. 中文章节标题识别。
2. 中文标点分段。
3. 中文数字与阿拉伯数字混合处理。
4. 中文姓名候选识别。
5. 别名与称谓识别。
6. 对白上下文识别。
7. GBK / GB18030 / UTF-8 编码识别。
8. 大文件增量处理。
9. 超长单文件处理。
10. 常见网文词汇适配。

中文姓名识别不能完全依赖分词，应结合：

- 高频短词
- 对白上下文
- “某某说”
- “某某问”
- “某某看着”
- “某某走进”
- 称谓词
- 用户手动确认

## 8. 核心能力模块

### 8.1 项目管理

功能：

- 创建项目。
- 导入本地目录。
- 导入单文件小说。
- 设置正文目录、设定目录、草稿目录。
- 设置题材模式。
- 设置存储模式。
- 设置 AI 模式。
- 删除项目索引但不删除原文件。
- 项目备份与迁移。
- 项目配置导入导出。

项目字段：

- 项目 ID
- 项目名称
- 作者名，可选
- 根目录
- 正文目录
- 设定目录
- 草稿目录
- 题材标签
- 启用模式包
- 存储模式
- AI 设置
- 创建时间
- 更新时间

### 8.2 文件扫描

功能：

- 初次全量扫描。
- 手动重新扫描。
- 文件监听。
- 增量扫描。
- 扫描队列。
- 扫描进度。
- 扫描失败恢复。
- 文件删除/重命名处理。
- 临时文件过滤。
- debounce。

扫描状态：

- Pending
- Running
- Completed
- Failed
- Canceled
- Skipped

建议 debounce：

```text
文件变化后等待 800ms - 2000ms。
如果期间继续变化，重新计时。
稳定后进入扫描队列。
```

### 8.3 文档与片段层

核心原则：

**原始正文必须有独立完整存储层。记忆图、摘要图、DAG、上下文压缩层不能成为唯一数据源。**

文档层负责：

- 保存文件路径。
- 保存文件 hash。
- 保存章节信息。
- 保存正文片段。
- 保存来源 offset。
- 支持从原文件回读。
- 支持重建索引。

片段层字段：

- document_id
- chunk_index
- start_offset
- end_offset
- content
- content_hash
- word_count
- source_path
- created_at

### 8.4 全文搜索

搜索范围：

- 正文
- 设定文档
- 人物卡
- 地点卡
- 物件卡
- 事件卡
- 伏笔卡
- 暗线卡
- 时间线
- 修改记录
- 冲突报告

基础搜索：

- 关键词搜索
- 多关键词搜索
- 章节过滤
- 人物过滤
- 地点过滤
- 文件类型过滤

高级搜索：

- 模糊搜索
- 别名搜索
- 相似片段
- 跨项目搜索，后续版本

搜索结果展示：

### 8.13 暗线与互文网络

暗线是伏笔系统的高级形态。

伏笔、暗线、互文关系：

```text
伏笔 = 文本明显给出的叙事承诺
暗线 = 反常、重复、微妙细节构成的潜在承诺
互文 = 两个或多个文本细节之间形成的呼应关系
```

暗线候选信号：

- 反常状态：大家都慌，某人平静。
- 重复意象：同一个物件反复出现。
- 非必要细节：对情节无立即作用但被具体描写。
- 异常动作：角色行为与当前动机不符。
- 特殊称呼：某人称呼突然变化。
- 信息缺口：有人知道不该知道的信息。
- 叙述停顿：用大量笔墨写一个暂时无用的东西。

系统不要直接判定，而是询问：

```text
这个反常细节是否要标记为暗线？
```

用户确认后，系统追踪：

- 是否回响
- 是否矛盾
- 是否遗忘
- 是否被删除
- 是否被提前揭示

## 9. 创作控制面

novellossless 的能力应组织为创作控制面，而不是零散功能。

### 9.1 控制面列表

| 控制面 | 解决什么失控 | 典型模式 |
|---|---|---|
| 连贯性控制 | 人物、设定、时间线矛盾 | 通用长篇 |
| 承诺控制 | 伏笔、暗线、悬念未回收 | 悬疑、长篇 |
| 情绪控制 | 爽点、虐点、甜点、高低潮 | 爽文、言情 |
| 阶梯控制 | 战力、财富、地位、权力成长 | 玄幻、都市 |
| 考据控制 | 历史制度、地名、官职、器物 | 历史 |
| 认知控制 | 角色知道/不知道什么 | 悬疑、权谋 |
| 风格控制 | 重复表达、文风漂移 | 所有人 |
| 反馈控制 | 本章说、评论、读者情绪 | 网文 |
| 版本控制 | 草稿演化、叙事漂移 | 精修型 |
| 套路控制 | 剧情单元复用、商业节奏 | 爽文 |

### 9.2 功能提炼原则

新增功能必须通过以下链条验证：

```text
作者类型
→ 最怕什么失控
→ 失控有没有可观测信号
→ 信号能否离线检测
→ 检测结果能否给出证据
→ 用户能否采取动作
→ 动作能否沉淀为记忆
```

跑不通的功能不进入核心。

## 10. 创作模式包系统

### 10.1 基本原则

novellossless 不应为所有作者提供同一套固定检查。  
它采用：

```text
核心能力 + 创作模式包
```

核心能力负责通用基础设施。  
模式包负责不同题材、不同写法、不同工作流下的专属规则和指标。

### 10.2 核心能力

所有项目共用：

- 项目导入
- 文件扫描
- 章节识别
- 全文搜索

## 11. 爽文模式

### 11.1 产品定位

爽文模式不是校对器，而是：

```text
节奏教练 + 爽点雷达 + 套路复用器
```

### 11.2 核心对象

| 对象 | 说明 |
|---|---|
| 主角阶梯 | 境界、财富、权力、身份、声望 |
| 对手阶梯 | 反派等级、压迫强度、势力背景 |
| 冲突单元 | 挑衅、羞辱、赌约、竞争 |
| 爽点事件 | 打脸、升级、爆装、身份揭露、反杀 |
| 压抑事件 | 被羞辱、被误解、被围攻、资源短缺 |
| 回报事件 | 获得资源、获得认可、击败对手 |
| 套路模板 | 拍卖会、炼丹、宗门大比、退婚、考核 |

### 11.3 爽点曲线

基础离线信号：

- 冲突词密度
- 胜利词密度
- 羞辱/嘲讽词密度
- 反转词密度
- 境界变化
- 金额变化
- 敌我身份变化
- “众人震惊”类反应词
- 评论密度，若导入本章说

输出示例：

```text
第 31 章：压抑 70 / 回报 20 / 爽点 35
第 32 章：压抑 60 / 回报 80 / 爽点 90
第 33 章：压抑 20 / 回报 15 / 爽点 25
第 34 章：压抑 15 / 回报 10 / 爽点 20

提醒：连续两章爽点密度下降。
```

### 11.4 打脸效果评估

打脸强度建模：

```text
打脸强度 = 对手压迫值 + 围观者数量 + 主角收益 + 身份反差 + 后果持续性
```

维度：

| 维度 | 信号 |
|---|---|
| 对手压迫值 | 对手身份、境界、势力 |
| 围观者数量 | 众人、全场、长老、宗主、百姓 |
| 主角收益 | 得宝、晋级、名声、职位、财富 |
| 身份反差 | 废物→天才，外门→长老，穷小子→贵客 |
| 后果持续性 | 反派退场、势力改变态度、剧情线推进 |

提醒示例：

```text
本次打脸效果偏弱：
- 对手等级低于上一次主要反派。
- 主角收益不明显。
- 围观反应较少。
- 结果没有改变后续局势。

建议：
- 提升对手身份。
- 增加公开场合。
- 让胜利带来资源或地位变化。
```

### 11.5 战力/财富/身份检查

检查：

- 战力倒退。
- 境界称呼不一致。
- 财富数字前后矛盾。
- 主角身份地位被错误降级。
- 势力范围变化无解释。
- 反派压迫强度边际递减。

示例：

```text
战力体系冲突：
主角已在第 42 章晋升为宗门长老，但第 45 章仍被描述为外门弟子待遇。

请确认：
- 第 45 章误写
- 主角隐藏身份
- 宗门内部刻意打压
- 标记为误报
```

### 11.6 套路模板库

支持用户把成功桥段保存为模板：

- 装逼打脸
- 拍卖会捡漏
- 炼丹打脸
- 宗门大比
- 退婚
- 身份揭露
- 强者降临
- 反派压迫后反杀

模板内容：

- 场景类型
- 参与角色
- 压迫阶段
- 反转阶段
- 回报阶段
- 读者反馈
- 可复用结构
- 禁止重复的表达

### 11.7 读者反馈导入

支持手动导入：

- 本章说
- 评论
- 段评
- CSV
- JSON
- 复制文本

第一阶段不做平台爬虫，避免合规风险。

导入后关联：

- 章节
- 段落
- 评论数量
- 高频词
- 情绪标签
- 爽点位置
- 吐槽位置

应用：

```text
你正在写类似“炼丹打脸”桥段。
上一部作品中，同类桥段在“揭炉反转”段落获得最高评论密度。
可参考该结构，但避免重复表达。
```

## 12. 历史考据模式

### 12.1 产品定位

历史考据模式不是自动查资料工具，而是项目内的历史考据沙盒。

它帮助作者在指定时代框架内写作，并允许用户管理架空例外。

### 12.2 历史框架

用户创建项目时设置：

```text
朝代：唐
具体时间：天宝三载
地点范围：长安、洛阳、河西
架空程度：低
考据严格度：高
```

系统形成约束：

```text
本项目默认以唐代天宝三载为历史框架。
所有官职、地名、食物、货币、器物、制度默认按该时间点检查。
```

### 12.3 本地知识包

历史考据必须支持本地知识包。

示例：

```text
history_tang.pack
├── 官职表
├── 地名表
├── 年号表
├── 货币表
├── 食物表
├── 服饰表
├── 器物表
├── 制度词表
└── 参考来源说明
```

用户可以导入自己的资料。

知识包字段：

- 条目名
- 类型
- 朝代
- 起始时间
- 结束时间
- 说明
- 来源
- 可信度
- 备注

### 12.4 考据检查

检查对象：

- 官职
- 地名
- 年号
- 货币
- 食物
- 服饰
- 器物
- 制度
- 称谓
- 交通方式
- 计量单位

提醒示例：

```text
可能时代穿帮：
当前项目设定为唐代天宝三载。
文本出现“番茄”。

处理方式：
- 改为当时可出现的食材
- 标记为架空设定
- 加入本书例外
- 修改时代框架
```

### 12.5 架空例外机制

用户可以选择：

- 接受提醒
- 标记为架空设定
- 加入本书例外
- 修改时代框架
- 忽略此类检查

架空例外必须进入项目规则，后续不重复报错。

### 12.6 暗线与互文

历史考据模式重点使用暗线网络。

系统可以发现：

- 反常描写
- 异常冷静
- 重复器物
- 特殊称谓
- 看似无用但具体的制度细节
- 某人提前知道信息

系统应询问是否标记为暗线，而不是直接判定。

## 13. 离线能力等级

### 13.1 L1：规则分析

不需要 AI。

能力：

- 章节识别
- 全文搜索
- 人物候选
- 地点候选
- 物件候选
- 显式伏笔候选
- 重复词句检测
- 基础时间线
- 基础人物属性冲突
- 文件改动记录
- 爽文基础指标
- 历史词表检查

### 13.2 L2：本地语义分析

不需要云端 AI，可使用本地算法或本地模型。

能力：

- 相似段落检测
- 重复意象检测
- 文风统计
- 简单语义聚类
- 卡片自动合并候选
- 相似伏笔候选
- 情绪曲线
- 叙事 Diff 基础版

### 13.3 L3：AI 增强分析

需要用户显式开启 AI。

能力：

- 隐性伏笔识别
- 复杂因果链分析
- 角色认知分析
- 世界观规则抽取
- 设定冲突解释
- 改稿影响解释
- 上下文包润色
- 叙事 Diff 高级解释
- 历史考据解释
- 爽点质量建议

默认启用 L1。  
L2 可作为本地高级能力。  
L3 必须由用户手动开启。

## 14. AI 增强系统

支持：

- OpenAI-compatible

- Anthropic-compatible

- 是否只发送摘要。

```text

每次调用前可预览发送内容。
```

## 15. 隐私与安全

### 15.1 默认隐私策略

默认：

- 不上传
- 不调用 AI
- 不扫描项目目录外内容
- 不读取剪贴板

- 不监听键盘
- 不监控其他应用

### 15.2 隐私中心

隐私中心展示：

- 当前是否离线模式
- 是否启用 AI
- 当前 AI Provider
- 最近 AI 请求
- 本地数据库路径
- 项目索引占用空间
- 删除项目索引
- 导出全部数据
- 清空 AI 请求记录
- 一键进入纯离线模式

### 15.3 本地存储模式

#### 标准模式

默认推荐。

本地数据库保存：

- 文档路径
- 正文片段
- 全文索引
- 来源证据
- 记忆卡
- 伏笔账本
- 冲突报告
- 改稿历史

优点：搜索和证据展示最快。  
适合大多数用户。

#### 隐私增强模式

本地数据库保存：

- 文档路径
- hash
- 索引
- 少量证据片段
- 记忆卡和报告

不保存完整正文缓存。

优点：减少数据库中的正文存量。  
缺点：部分功能需要回读原文件。

#### 最小索引模式

本地数据库保存：

- 文件路径
- offset
- hash
- 索引元信息
- 用户确认的卡片

展示来源时尽量回读原文件。

优点：隐私最强。  
缺点：搜索、报告和改稿分析能力受限。

### 15.4 数据删除

用户可以：

- 删除单个项目索引
- 删除某张卡片
- 删除某条分析记录
- 删除所有 AI 请求记录
- 删除全部本地数据
- 重建索引

删除索引不得删除用户原始小说文件，除非用户明确选择。

### 15.5 备份与迁移

用户可以：

- 导出项目索引
- 导出记忆卡
- 导出伏笔账本
- 导出冲突报告
- 导出上下文包
- 备份整个项目数据库
- 在新设备导入备份

备份格式：

- JSON
- Markdown
- SQLite 备份包
- 压缩包

备份不得默认包含原始小说正文，除非用户明确选择。

## 22. 验收标准

### 22.1 离线验收

断网情况下，用户应能：

- 打开应用

- 扫描 TXT / Markdown
- 搜索全文

- 查看伏笔账本
- 查看冲突报告
- 查看修改历史
- 生成上下文包
- 导出报告

### 22.2 隐私验收

默认状态下：

- 不要求登录
- 不调用 AI
- 不上传正文
- 不读取项目目录外文件
- 不读取剪贴板
- 不截屏
- 不监听键盘
- 可以删除本地索引
- 可以查看数据库路径

### 22.3 记忆验收

给定一部 20 万字测试小说，系统应能：

- 识别主要人物
- 识别主要地点
- 识别重要物件
- 识别显式伏笔
- 生成章节索引
- 生成基础时间线
- 生成上下文包

### 22.4 冲突验收

测试小说中故意包含：

1. 人物眼睛颜色冲突。
2. 父亲死亡 / 失踪冲突。
3. 重要物件持有人冲突。
4. 魔法规则冲突。
5. 长期未回收伏笔。
6. 重复雨夜描写。

系统应至少发现其中 4 类，并给出来源证据。

### 22.5 改稿验收

用户修改前文关键事实后，系统应能：

- 记录改稿事件
- 更新 content hash
- 更新相关卡片
- 检测受影响后文
- 生成影响提示
- 创建修订任务

### 22.6 爽文模式验收

测试文本中包含：

1. 主角境界晋升。
2. 后文错误降级。
3. 连续三章无明显冲突。
4. 打脸场景对手等级下降。
5. 财富数字前后不一致。

系统应至少发现其中 3 类，并给出来源证据。

### 22.7 历史考据模式验收

给定本地知识包和历史框架，测试文本中包含：

1. 不合时代的官职。
2. 不合时代的食物。
3. 地名时代错位。
4. 人物提前知道不该知道的信息。
5. 作者标记架空例外。

系统应至少发现前 4 类中的 3 类，并在用户标记例外后不重复报错。

## 26. 最终产品愿景

novellossless 最终要成为长篇小说作者的第二记忆和创作控制台。

它不替作者创作，不抢作者的表达权，而是在长篇创作中承担最痛苦、最机械、最容易出错的部分：

- 记住人物。
- 记住伏笔。
- 记住设定。
- 记住暗线。
- 记住改动。
- 记住时间线。
- 记住每个角色知道什么。
- 记住读者在哪里兴奋、在哪里失望。
- 记住每一次草稿变化带来的叙事漂移。

当作者写到几十万字之后，novellossless 应该能回答：

```text
这个人物之前出现过几次？
这个伏笔有没有收？
这条规则有没有被破坏？
这个角色此时知道真相吗？
这次修改会影响后面哪里？
最近三章是不是节奏变平了？
这段考据是否可能穿帮？
我要写下一章，需要带上哪些上下文？
```

novellossless 的终极价值不是“帮你写得更多”，而是：

> 帮你守住长篇创作中最容易失控的部分。
